## C# 编码规范

### 文件结构规范

**成员排序优先级**：

1. **按访问修饰符排序**：公开 → 保护 → 私有 SerializeField → 私有
2. **按成员类型排序**：数据结构 → 构造函数 → 字段 → 属性 → 函数
3. **按功能逻辑排序**：按执行顺序排序、按成对关系排序、按同类变体排序

### 命名规范

- **基本风格**：驼峰命名法，怕长不怕短，要容易理解，不要用下划线
- **倒序域名化**：如 `DbTableServer`
- **大小写**：非常量私有成员小写开头，公开成员大写开头
- **类型格式**：
  - 接口：`IInterface`
  - 泛型：`TGeneric`
  - 布尔：`IsDoorOpening`
  - 事件：`DoorOpen`（执行前）、`DoorOpening`（执行中）、`DoorOpened`（执行后）
  - 函数：`OpenDoor`

### 代码设计原则

- 字段永远是私有的，除了只读字段
- 多个 if 时采用级联而非嵌套
- `!bool` 仅用于取反，逻辑判断用 `bool == false`
- switch/if 无法确定最大数量时改用继承重写、委托
- 函数过长时拆成局部方法
- 代码关系层级分明，形成树形结构，不要越级操作
- 出现复制粘贴时考虑封装避免重复

### 注释与文档

- 变量名应自解释，好的命名胜过注释
- 只在复杂逻辑或特殊用况时添加注释
- 不要写无意义的 XML 文档注释

### 异常与验证

- **面向程序员 of API**：让异常自然抛出，程序员应自行处理错误
- **面向用户的代码**：适当添加验证 and 异常处理
- 明显不应该为空的参数：不需要判空
- 可能为空的业务场景：应该验证

### 现代语法优先

- 使用所有能提升可读性和简洁性的语法糖
- 例如：using 声明、模式匹配、表达式主体、目标类型 new 等
- 新语法应该让代码更清晰，而非炫技

### 返回值设计

- 不会失败的操作：不要返回 bool/int 等状态码
- 可能失败的操作：返回有意义的结果或状态
- 避免 `return true` 这种永远成功的假象

### 扩展性设计

- 不要预先过度设计（如到处加 virtual）
- 明确需要扩展时才使用 virtual/abstract
- 保持 YAGNI 原则（You Aren't Gonna Need It）

### 性能意识

- 避免明显的性能问题（如减少内存分配，复用对象结果）
- 不要为微优化牺牲可读性
- 在性能关键路径上才需要极致优化

## Unity 特定规范

### 初始化规范

- Awake时完成自身初始化，Start时可以引用其他模块
- Awake 和 Start 期间可以修改参数，按情况配备初始化函数
- 尽可能利用序列化引用物体而非运行时绑定，使引用关系在面板可见

### 插件/模块文件夹结构

```text
- Sources/   # 源代码（核心实现、基础脚本、原始素材等）
- Products/  # 产物（完整预制体、可直接挂载的功能脚本等）
- Samples/   # 产物（Demo 场景、测试脚本等）
```

**优势**：清晰分离核心代码和演示产物，便于理解和导出

## 技术积累与避坑指南

### 1. 流式 XML 执行器 (XmlStreamExecutor)
- **并发控制**：使用 `Channel<T>` 将所有操作（Feed, Flush, Reset）排队执行。这能有效防止 `Reset` 或 `Flush` 在后台解析过程中强行操作标签栈导致的 `Stack empty` 异常。
- **状态重置**：`Reset` 逻辑应先清空当前指令队列，再发送重置指令，确保状态切换是原子且干净的。

### 2. WPF 交互与位移
- **多显示器支持**：边界检测应使用 `SystemParameters.VirtualScreen` 相关属性，而非 `WorkArea`，以防止窗口在副屏位移时被非法“吸回”主屏。
- **逻辑坐标同步**：除了程序位移，手动拖拽（`LocationChanged`）也必须实时同步逻辑终点坐标。否则下次程序位移会基于旧的逻辑坐标计算偏移量，导致位移量离奇。

### 3. Live2D 与眼神控制
- **状态护盾 (Gaze Guard)**：在说话等需要正视前方的状态下，应在 `mousemove` 监听器顶部增加状态判断并立即 `return`。严禁在监听器内频繁调用 `resetGaze`，否则高频的参数拉扯会造成视觉上的剧烈抖动/卡顿。

### 4. 视觉语义识别 (Vision Semantic Analysis)
- **非 AI 依赖特征提取**：利用 OpenCV 直接提取亮度、复杂度（边缘密度）和主色调。这种物理特征标签化方法能有效替代高成本的多模态 AI 接口，为下游 LLM 提供充足的视觉线索喵。
- **多特征级联识别**：级联分类器（Haar Cascades）不仅局限于人脸，扩展到眼睛、猫脸等特定特征能极大地增强真央的拟人化程度。

### 5. 语音交互响应性 (Speech Responsiveness)
- **静音自动断句 (Silence Auto-Finalization)**：在 `LocalSpeechRecognizer` 中通过监控 `PartialResult` 的变化时间差来检测静音。一旦静音超过阈值（如 800ms），立即强制调用 `Result()`。这比等待模型自带的弱响应要快得多，能显著降低用户感知的交互延迟。

### 6. Demo 标准化测试框架 (DemoSuite)
- **开箱即用 (Out-of-the-box)**：通过 `DemoSuite.InitializeAsync(character)` 一键完成环境初始化（Storage, Config, ChatActivity）。
- **全链路追踪 (Traceability)**：订阅 `ChatHistoryAdd` 事件而不是简单的 `ChatSent/Received`。这样做可以捕获所有被插件（Plugins）注入的系统提示词（System Prompts）和背景指令，实现真正的“两边收发情况”透明化记录喵。
- **UI 色彩规范**：
    - `SYS` / `TOOL`：深黄色 (DarkYellow)，用于系统指令和插件日志。
    - `SENT`：青色 (Cyan)，用于主动发出的消息。
    - `RECV`：绿色 (Green)，用于接收到的原始数据或 AI 结果。

### 7. XML 流式解析转义 (XmlStreamParser Escape)
- **转义符支持**：在 `XmlStreamParser` 中引入了 `\` 作为转义符。
- **使用场景**：`\<>` 会被视为普通文本 `<>` 输出，而非 XML 标签。`\\` 会输出单个 `\`。这在需要输出包含标签符号的原始文本时非常有用。
### 8. 永久循环后台任务 (Perpetual Background Tasks)
- **使用 `async void`**：对于在对象生命周期内始终循环执行且不返回结果的私有后台函数（如 `LoopProcessInput`），优先使用 `async void` 而非 `Task`。
- **设计理由**：因为这类函数无法被有效 `await`（永远等不到结果），若返回 `Task` 会误导调用者。通过 `async void` 并在内部进行全量 `try-catch` 处理，可以确保任务的独立性与安全性，同时简化外部调用逻辑。

### 9. 分层记忆存储 (Hierarchical Memory Storage: Frontier & Archive)
- **核心模式**：采用 **Frontier (活跃快照)** 与 **Archive (归档批次)** 分离的架构。
- **快照持久化 (Frontier)**：每一轮对话（OnChatOver）触发一次 `SaveFrontier`，将当前内存中所有层级（L0 - Ln）的活跃记录全量保存为 `{characterId}/Frontier.json`。这保证了会话意外中断后能完美恢复。
- **批次归档 (Archive)**：当 L0 达到阈值触发压缩时，将这组原始记录保存到 `Archives/{summaryId}.json`。摘要记录的 `Id` 作为文件名，实现了稳健的溯源。
- **上下文还原**：通过 `RefreshChatHistory` 逻辑将分层记忆动态映射回 `ChatHistory`。对于 Level 0 记录，应通过文本拆分（"用户：" / "回复："）还原 User/Assistant 角色，而非简单的单条注入。
